# בוט ניתוב פניות — תרחישי בדיקה ידנית ל-POC

מסמך זה מסביר איך לבדוק ידנית את ה-POC ולוודא שתהליך המשתמש עובד כמתוכנן. בכל תרחיש מופיע הקלט המדויק שיש להזין באפליקציה, הסיווג הצפוי, ומה אמור להופיע על המסך.

> **חשוב:** הסיווג הראשוני מבוסס כעת על חוקים דטרמיניסטיים ומילות מפתח. הוא זמני. המטרה היא להדגים את חוויית המשתמש ואת ההיגיון התפעולי לפני החלפת המנוע ב-LLM אמיתי.

## דרך בדיקה כללית

1. התחברות עם משתמש הדמו: `matan@rashi.org` / `Demo!2026`.
2. **צור פנייה חדשה** מהלוח הראשי.
3. הזן את שדות התרחיש (מחלקה, ספק, סכום, תיאור, קבצים). אפשר להעלות כל קובץ דמה.
4. לחץ **נתח פנייה**.
5. ודא שהסיווג המתקבל זהה לערך **route צפוי** של התרחיש.
6. בתרחישי `legal_review`, לחץ **המשך להשלמת פרטים** וודא שהשדות מולאו מראש לפי המסומן.
7. במסך אישור השליחה, לחץ **📄 הורד מסמך Word לבדיקה משפטית** וודא שמתקבל קובץ DOCX עברי RTL.
8. סיים בלחיצה על **שלח לבדיקה משפטית** וודא שמופיע מסך האישור.

## סדר עדיפויות סיווג

1. `missing_info` (רק כשהתיאור ריק לחלוטין)
2. `legal_review` (טריגרים משפטיים, תנאי ספק, סכום ≥ 200K, או מענק עם טריגר משפטי)
3. `grant`
4. `supplier_registration`
5. `insurance_required`
6. `general_terms` (ברירת מחדל)

## תרחיש 1 — תנאי התקשרות כלליים (general_terms)

| | |
|---|---|
| **מיזם / מחלקה** | מערכות מידע |
| **שם ספק** | מחשוב ישיר |
| **סכום משוער** | 18000 |

**תיאור הפנייה:**
> אנחנו צריכים להזמין שירותי התקנה של 12 עמדות מחשב חדשות במשרד התל-אביבי. ההצעה כוללת תכולה, סכום ולוח זמנים. עבודה חד-פעמית, ללא תנאי תשלום מיוחדים או דרישות נוספות.

**Route צפוי:** `general_terms`

**למה הסיווג הזה נבחר:**
- אין מילות מפתח של בדיקה משפטית, ביטוח, רישום ספק או מענק.
- הספק *מחשוב ישיר* מסומן ב-DB כ-`registered`.
- ההצעה "נקייה" — תכולה / סכום / לו"ז בלבד.
- העבודה חד-פעמית ובסכום נמוך מ-200,000 ₪.

**מה המערכת אמורה להציג בסיכום:**
- המלצה: *תנאי התקשרות כלליים / ספק במאגר*.
- מסר ראשי: *"ניתן להתקדם על בסיס תנאי ההתקשרות הכלליים של קרן רש״י, בכפוף לכך שהספק במאגר, קיימת הצעת מחיר נקייה, ותונפק הזמנת רכש חתומה לפי נוהל הקרן."*
- כפתור ראשי: **הצג תנאי התקשרות כלליים** (יוריד את `rashi-general-terms.docx`).
- באזור **מה המערכת הבינה**: אופי ההתקשרות = "עבודה חד-פעמית", סטטוס ספק = "רשום במאגר 2026".

**שדות שאמורים להתמלא אוטומטית:** לא נכנסים ל-intake המשפטי בתרחיש זה.

## תרחיש 2 — רישום ספק במאגר (supplier_registration)

| | |
|---|---|
| **מיזם / מחלקה** | משאבי אנוש |
| **שם ספק** | קייטרינג הבית |
| **סכום משוער** | 8000 |

**תיאור הפנייה:**
> נדרשת הזמנת ארוחות עבודה חד-פעמית למשרד. הספק מוכר לנו אבל אני לא בטוח אם רשום במאגר 2026.

**Route צפוי:** `supplier_registration`

**למה הסיווג הזה נבחר:**
- *קייטרינג הבית* מסומן ב-DB כ-`not_registered`.
- הביטוי *"לא בטוח אם רשום"* + *"מאגר 2026"* טריגר רישום ספק.
- "ארוחות" יכול להפעיל ביטוח, אך לפי סדר העדיפויות `supplier_registration` קודם ל-`insurance_required`.

**מה המערכת אמורה להציג בסיכום:**
- המלצה: *רישום ספק במאגר*.
- כפתור ראשי: **פתח קישור רישום ספק (SAP)** — קישור אל הדף הדמה ב-`https://sap-rashi.example.org/...`.
- כפתור משני: **צור הודעה לספק** — מציג טקסט מוכן להעתקה.
- הערה: *"כחלק מהרישום הספק חותם על תנאי ההתקשרות הכלליים של הקרן."*

**שדות שאמורים להתמלא אוטומטית:** רק אם המשתמש בכל זאת לוחץ "העבר לבדיקה משפטית".

## תרחיש 3 — נדרש אישור ביטוח (insurance_required)

| | |
|---|---|
| **מיזם / מחלקה** | משאבי אנוש |
| **שם ספק** | ODT ישראל |
| **סכום משוער** | 35000 |

**תיאור הפנייה:**
> מתכננים יום גיבוש לעובדים בצפון עם פעילות ODT הכוללת חבלים ואתגרי שטח, וטיול ג'יפים. כולל הסעות ארוחת צהריים מהמלון.

**Route צפוי:** `insurance_required`

**מה המערכת אמורה להציג:**
- המלצה: *תנאי התקשרות + השלמת אישור ביטוח*.
- מסר: *"ניתן להתקדם בתנאי ההתקשרות הכלליים, ובמקביל יש להשלים אישור ביטוח לפי סוג השירות..."*.
- כפתור ראשי: **הצג תנאי התקשרות כלליים**.
- תיבת אזהרה צהובה עם רשימת הקטגוריות (פעילות אתגרית / הסעות / אוכל / לינה).
- אם פרטים חסרים — הקלסיפייר עשוי להחזיר עד 3 שאלות הבהרה.

## תרחיש 4 — בדיקה משפטית: ייעוץ מתמשך (legal_review)

| | |
|---|---|
| **מיזם / מחלקה** | הנהלה |
| **שם ספק** | יועצי אסטרטגיה ABC |
| **סכום משוער** | 90000 |

**תיאור הפנייה:**
> אנחנו רוצים להתקשר עם יועץ אסטרטגי לתהליך מתמשך של ליווי הנהלה למשך חצי שנה בסכום של 90,000 ₪. קיימת הצעת מחיר. הסכם חדש.

**Route צפוי:** `legal_review`

**שדות שאמורים להתמלא אוטומטית ב-intake:**
- מטרת ההתקשרות (משפט ראשון)
- סוג ההתקשרות (הערכה): *רכישת שירות / מוצר*
- תפקיד כל צד: *"קרן רש"י מזמינה שירות / מוצר מ-יועצי אסטרטגיה ABC..."*
- חדש / המשך: *חדש*
- שם הצד השני, סכום, לוח זמנים *חצי שנה*
- הספק כבר נבחר: *כן*
- קיימת הצעת מחיר: *כן*
- ניקיון ההצעה: *נקייה* (לא זוהו תנאי ספק חריגים)

**במסך אישור — לחץ "📄 הורד מסמך Word"** וודא שמתקבל קובץ DOCX עברי RTL עם 10 הסעיפים.

## תרחיש 5 — בדיקה משפטית: תנאי ספק + IP

| | |
|---|---|
| **מיזם / מחלקה** | חינוך |
| **שם ספק** | יועצי אסטרטגיה ABC |
| **סכום משוער** | 120000 |

**תיאור הפנייה:**
> פיתוח תוכן לימודי דיגיטלי. הספק דורש לחתום על ההסכם של הספק, עם תנאי תשלום של מקדמה 50% מראש, שמירת בעלות על הקוד, וזכויות פרסום על תוצרי הפרויקט. הרשות המקומית היא שותפה לתוכנית.

**Route צפוי:** `legal_review`

**למה:**
- *"ההסכם של הספק"*, *"תנאי תשלום"*, *"מקדמה"*, *"שמירת בעלות"*, *"זכויות פרסום"* — כולם טריגרים של תנאי ספק → ההצעה אינה נקייה.
- *"הרשות המקומית"* — שותף ציבורי.
- מסווג כ-legal_review בהפרש משמעותי.

**Prefill חשוב:**
- ניקיון ההצעה: *כוללת תנאי ספק*.
- תנאי ספק שזוהו: רשימת תיוגים אדומה תופיע ב-Card 2.
- שותפים: *כן*.
- IP / זכויות יוצרים: *כן*.
- סוג ההתקשרות (הערכה): *שיתוף פעולה* או *רכישת שירות / מוצר* (תלוי בניסוח).

## תרחיש 7 — מסלול מענק (grant)

| | |
|---|---|
| **מיזם / מחלקה** | חינוך / חברה |
| **שם ספק / צד שני** | עמותת לדוגמה |
| **סכום משוער** | 75000 |

**תיאור הפנייה:**
> קרן רש"י מעוניינת להעביר מענק של 75,000 ₪ לעמותת "לדוגמה" עבור הפעלת תוכנית קיץ לילדי השכונה. הפעילות היא של העמותה, לא של הקרן.

**Route צפוי:** `grant`

**מה צריך לראות:**
- המלצה: *מסלול מענק*.
- מסר: *"...יש להשתמש במאסטר כתב התחייבות לקבלת מענק ולהשלים את חבילת המסמכים הנדרשת."*
- כפתור ראשי: **פתח מאסטר כתב התחייבות למענק** (יוריד את `rashi-grant-master.docx`).
- ב-intake המשפטי — מופיע **Card "מסמכי מענק"** עם 8 מסמכים: אישור מנכ"ל, בקשת מענק, טופס מענק, תקנון, ניהול תקין, סעיף 46, ניכוי מס, אישור רו"ח.

**Prefill חשוב:**
- סוג ההתקשרות (הערכה): *מענק / תמיכה*.
- תפקיד כל צד: *"קרן רש"י מעניקה מענק ל-עמותת לדוגמה לטובת פעילות שלהם..."*.

## תרחיש 8 — מענק עם רכיב משפטי

| | |
|---|---|
| **מיזם / מחלקה** | רווחה |
| **שם ספק** | עמותת לדוגמה |
| **סכום משוער** | 250000 |

**תיאור הפנייה:**
> תמיכה כספית לעמותה לטובת פרויקט משותף עם משרד ממשלתי. הסכום מתחלק בין הקרן והמשרד הממשלתי. כולל איסוף תעודות זהות של משתתפים.

**Route צפוי:** `legal_review` (לא `grant`!)

**למה:**
- יש טריגרים של מענק (*תמיכה*, *עמותה*) אבל גם של legal_review (*משרד ממשלתי*, *תעודות זהות*, סכום ≥ 200K).
- לפי סדר העדיפויות, legal_review גובר.
- במסר יופיע: *"פנייה זו כוללת גם רכיב מענק; הבדיקה המשפטית תכלול את שני ההיבטים."*
- תגית *מענק* תופיע יחד עם תגית *בדיקה משפטית*.

- [ ] תרחיש 7: grant — Card "מסמכי מענק" עם 8 שאלות yes/no.
